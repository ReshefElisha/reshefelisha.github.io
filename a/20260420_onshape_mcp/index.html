Teaching Claude CAD skills. Onshape MCP and visual reasoning tools

April 20, 2026 · Reshef Elisha

Working on hardware, I've been able to supercharge my software thanks to agentic tools. I haven't been able to do this for hardware until now. Claude Opus 4.7 came out last week with advanced visual reasoning skills. This is the first model that, when combined with its harness, can do actual productive mechanical engineering and CAD.

I've shipped the result as Jarvis Onshape MCP, a Claude Code plugin that drives Onshape. You install it with one command. Here's the monitor arm it built for me yesterday:

Monitor arm build, 10× speed.

That's four parts. A desk-clamp base with a post, two arm segments with cross-rib structural channels (which admittedly still need a bit of work), and a monitor pivot with a VESA-compliant mounting plate. I gave Claude a reference sketch and a paragraph-and-a-half description. It produced a structured feature tree, showed me the plan, and then built it. Start to finish in about an hour of real time.

I've been building in CAD for years. This is the first time an AI tool has felt like it was working with me on a design, instead of flailing miserably trying to do mental visualization it was never built to do.

The experiment I didn't expect to run

I went in assuming the hard problem was CAD execution. I assumed that driving the kernel, using native CAD features, keeping track of which face to sketch on, keeping track of geometry through tool calls, that was going to be the grind.

I was wrong. With a decent (and continuously-improving) MCP, CAD functions are just the same as code.

The hard part is that Claude is (still) blind.

To figure out where Claude was actually getting stuck, I built an eval harness. 27 reference parts across three difficulty tiers. A layered grader that compares the agent's output STEP against a ground-truth STEP on six geometric axes: volume, bounding box, topology signature, Boolean intersection-over-union, Chamfer distance between surfaces, all composited into a single score in [0, 1]. The grader is rotation and translation invariant, so "correct part in a different world frame" still counts as correct. The usual boring stuff.

Then I spent two days running variants. Prompt tweaks, new MCP tools, dataset rebalancings. Everything I could think of that might make Claude read engineering drawings better.

Almost nothing worked. One variant of "telling the agent to list features from the reference render before building" moved the score from 0.155 to 0.193 on medium-difficulty parts. A +0.04 improvement, four times above the noise floor, but still barely a scratch. The other seven variants either didn't move the needle or actively regressed.

That's when I tried the experiment that actually settled the question.

The 2×2

I decided to take the hardest brief Claude could kind of do (scoring around 0.5 on) and write the spec by hand. A plain-English paragraph describing every feature. Then feed it to the build phase of the agent, skipping the vision decomposition entirely.

Here's what happened:

Model Mania 2025 (simpler) Model Mania 2021 (harder)
Autonomous vision 0.533, 147 turns, 40 min 0.000 (never exported), 151 turns
Handwritten spec 1.000, 57 turns, 7 min 0.615, 102 turns, 27 min

Same model. Same CAD tools. Same grader. The only difference is who wrote the spec.

Model Mania 2025 with a clean human description: perfect score on every grader layer. Volume off by 0.007% from ground truth. Built in seven minutes.

The autonomous version of the same brief, with 147 turns of self-correction, topped out at 0.533. Not because it couldn't build what it described but because it described the wrong thing. For example, it saw a pocket on the front face and modeled it as a simple rounded rectangle. The actual pocket outline was the part's silhouette offset inward by 10mm, cutting around two holes. Claude simplified it confidently and shipped, wrong. Any human reading the drawing could give the right shape, but Claude's vision is still blurry.

Model Mania 2021, the harder part: hand-written description scored 0.615, and it was far from my best description (look at the part yourself and try to describe it, it's tough!) The autonomous version scored 0.000: it built something, but ran out of turn budget before it managed to export it.

What this means

The bottleneck for AI-driven CAD isn't the kernel. It isn't the tool surface. It isn't even the prompt scaffolding, which I spent two days tweaking. It's image understanding. Opus 4.7 can read a reference image well enough to understand "this is a bracket with some holes," but can't yet be trusted to infer the specific geometry — the polarity of a feature (boss or pocket?), the derivation of a shape (offset from what?), the scale when the callouts are small.

What it can do is execute engineering geometry reliably when given a clean spec. Once the features are described unambiguously, the build is mechanical (pun intended).

That's a good result, actually. It means the plugin has a clear niche: not "autonomous CAD" (we're not there yet), but "CAD copilot driven by design intent." You describe what you want in plain English. Claude produces a feature tree. You sanity-check it, maybe correct a mis-read, and then it builds. The workflow replaces keyboard shortcuts and feature-tree surgery with typing and verbal iteration. If you already know what you're building, Claude is remarkably fast at building it. As usual with early AI tools, the first step to building something is being able to describe what you want to build (🙄)

The takeaway

From the 2×2 experiment onwards, I focused on giving Claude glasses. All the tools I had it build were focused on improving model inspection, allowing it to render, describe in words, zoom, crop, etc... This meaningfully helped it catch its own mistakes while building.

What's in the plugin

About 60 MCP tools covering sketches, extrudes, fillets, patterns, mates, parametric variables, and a FeatureScript escape hatch for the weird stuff. Everything returns structured truth: what feature got built, what warnings the regenerator raised, what the next likely move is. No silent failures.

Two skills ship with it:

  • An Onshape CAD skill that encodes the protocols that keep CAD builds from silently failing. Unit conventions, coordinate frames, render-first and entity-first workflows, iteration discipline.
  • A vision-decomposition skill that walks the model through reading a reference image before building. Overview → cache → zoom into each feature → structured output. It's not magic — my benchmark showed it doesn't close the vision gap — but it makes the gap visible. You see the feature tree before Claude commits to it. You can fix the mis-reads that matter before burning turns on a wrong build.

The thing I want to say about this

In hardware, we're watching the software world get eaten by AI and are counting the days until we meet the same fate. I think we're there. Last month I would have said we'd need a dedicated vision model trained on engineering drawings. Today I'm publishing a plugin that an engineer can install in thirty seconds and use productively for real design work, as long as they're willing to type a paragraph.

The gap between "it needs a massive training cluster" and "it's doable in a plugin" is shorter than it feels. And the direction is obvious once you look: the capability bottleneck isn't where we expect. CAD kernels are solved. Prompt engineering is a distraction. The hard problem is whether the model can look at an engineering drawing and know what it's looking at. I wouldn't be surprised if there are dedicated vision models that can (or soon will) create the right descriptions for Claude even better than I can.
